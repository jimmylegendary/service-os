# PRINCIPLES.md

## Core product principles

### 1. Reduce cognitive burden
Products should reduce the amount of active mental overhead carried by the user.
They should not simply become better external storage for anxiety.

### 2. Present-moment action matters more than perfect organization
The goal is not exhaustive management.
The goal is helping the user act on what matters now.

### 3. Philosophy is expensive to change
Early philosophy matters.
Product essence, problem framing, and what a service should never become must be clarified early.

### 4. MVP is sacred
A product should begin with the smallest version that can validate the real thesis.
Extra features are not progress if they blur the thesis.

### 5. Roadmaps are directional, not rigid
Think broadly.
Plan intentionally.
But expect change from reality.

### 6. Human-AI collaboration is a first-class design surface
Products should increasingly support coordination between:
- human intent
- machine interpretation
- delegated execution
- feedback loops

### 7. Flows over lists
Systems should be understandable as flows and state transitions, not flat piles of items.

### 8. Build reusable patterns
Service-specific work is valuable, but cross-service patterns are compounding assets.

## Build principles

### 1. No waterfall
Never optimize for complete speculative design over practical learning.

### 2. No blind hacking
Never skip the thinking required to define a coherent MVP.

### 3. Fractal decomposition everywhere
Every large problem should be reducible to nested `input -> function -> output` flows.

### 4. Logs should teach the future system
Every meaningful project should leave behind reusable knowledge.

### 5. Automation should be progressive
Do not aim for total automation first.
Start by automating repeatable parts of:
- planning
- implementation
- testing
- deployment
- feedback triage
